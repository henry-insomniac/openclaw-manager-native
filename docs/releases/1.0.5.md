# OpenClaw Manager Native 1.0.5

发布日期：2026-03-11

## 这版改了什么

- 诊断中心拆成“自动轻刷新 + 手动强刷新”两条路径。自动刷新优先复用短期缓存，不再每轮都重跑整套 OpenClaw 检查。
- 原生壳去掉重复全量轮询，普通页面只拉 manager；诊断页和手动刷新才会拉完整 support 摘要，后台空闲 CPU 明显下降。
- 安装版启动链修掉了清理历史 backend 时的 pipe 阻塞死锁，旧的 bundled daemon / Node API 残留也会在启动时一起清掉。
- 新增原生生命周期日志，定位安装版启动和后台驻留问题不再是黑盒。

## 相对 1.0.4 的性能优化明细

### 1. 前端刷新模型从“总是全量”改成“按需拉取”

- `NativeStore` 新增 `NativeRefreshScope`，把刷新拆成 `full` 和 `managerOnly`。
- 移除了启动时那次额外的立即全量刷新，避免 app 刚起来就重复打一轮 `manager + support`。
- 移除了菜单栏原来每 `20s` 一次的全量轮询。
- 保留的一条 `30s` 定时刷新也不再总是走全量：
  - 普通页面只拉 `/api/openclaw/manager`
  - 只有诊断页才拉完整 support 摘要
- 进入诊断页时再补一次静默 full refresh，而不是让所有页面都长期承担诊断成本。
- 登录流程结束后也只补一轮静默 full refresh，不再触发无差别全量刷新。

### 2. support 接口增加“缓存快路径”和“fresh 强刷”语义

- 原生壳默认轮询 `/api/support/summary`，优先命中 daemon 缓存。
- 只有用户手动刷新时才调用 `/api/support/summary?fresh=1`，明确要求重跑深检查。
- 这意味着 `1.0.5` 真正优化的是“高频常驻路径”，而不是把每一次深检查都硬压成毫秒级。

### 3. daemon 端对 support 各个子模块做了分层缓存

- `supportSummary`：`10s`
- `gateway status`：`45s`
- `recent events`：`45s`
- `environment summary`：`2min`
- `maintenance summary`：`2min`

这样做的效果是：

- 高频轮询不会每次都重跑 `openclaw status --json`
- 不会每次都重扫 gateway / watchdog 日志尾部
- 不会每次都重跑 `route / scutil / ifconfig / pmset`
- 不会每次都重跑 `openclaw config validate` 和 `gateway status --deep`

### 4. profile 发现增加短 TTL 缓存

- daemon 新增 `stateDirCacheTTL = 45s`
- `discoverStateDirs()` 不再每次都递归扫 OpenClaw 根目录寻找 `.openclaw-*`
- 只有 profile scaffold 发生变化时才主动失效缓存

这一步主要优化了 `manager` 路径，降低了目录扫描在频繁刷新下的重复开销。

### 5. 安装版启动链修掉了一个真实的性能/稳定性阻塞点

- 原生壳此前在清理历史 bundled backend 时会调用 `ps -axo pid=,command=`
- `runCommand()` 旧实现是先 `waitUntilExit()` 再读 pipe，遇到大输出时会把 pipe 塞满并死锁
- `1.0.5` 把 stdout / stderr 改成并发读取，彻底消掉这个启动卡死点

这个修复不只是“更稳”，它直接解决了安装版卡在启动链上的真实阻塞。

### 6. 启动前会清理历史 bundled backend，避免旧进程持续施压

- 启动时会扫掉旧的 bundled `openclaw-manager-daemon`
- 也会扫掉老架构残留的 `node ... apps/api/dist/server.js`

这一步主要解决升级后旧后台进程残留、重复占用端口、继续吃 CPU 的问题。

### 7. 原生 app 生命周期收紧，减少后台异常回收和黑盒排障成本

- 显式禁用 automatic termination / sudden termination
- 增加 `native.log` 生命周期日志

这部分不是直接把接口耗时降到更低，但它减少了“后台被系统回收后反复重启、再触发重检查”的隐性性能损耗，也让后续排障更可控。

## 实测结果

- 安装版空闲 35 秒后：`OpenClawManagerNative` 与 `openclaw-manager-daemon` 都为 `0.0% CPU`
- `/api/openclaw/manager`：约 `0.074s`，重复请求约 `0.024s`
- `/api/support/summary`：缓存命中约 `0.005s`，重复请求约 `0.001s`
- `/api/support/summary?fresh=1`：约 `3.739s`

可以把 `1.0.5` 的收益理解成：

- `1.0.4` 已经把“接口本身太慢”从十秒级压到了秒级
- `1.0.5` 继续把“高频刷新路径”和“后台空转成本”压掉
- 仍然保留了手动 `fresh` 深检查，所以深检查本身还是秒级，不是毫秒级
